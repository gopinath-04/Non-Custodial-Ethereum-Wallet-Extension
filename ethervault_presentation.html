The Centralized Web3 Problem

  • "Not Your Keys, Not Your Coins": The defining lesson of the last crypto cycle.
  • Exchange Collapses: The FTX fallout exposed the dangers of trusting centralized custodians with user funds.
  • Security Vulnerabilities: Centralized databases are honeypots for hackers, resulting in billions lost annually.
  • Loss of Sovereignty: Users are blocked from their own assets during maintenance or bank runs.

The Solution: True Self-Custody

Custodial (Banks/CEX)

A third party holds your private keys. You have an "IOU" for your money, not the money itself. Vulnerable to censorship and insolvency.

Non-Custodial (EtherVault)

You hold the keys. The wallet is just an interface to the blockchain. Even if EtherVault disappears, your funds exist on-chain.

4. Blockchain Provider: JSON-RPC connection to Ethereum Mainnet/Testnet.

Deep Dive: Manifest V3

Legacy (Manifest V2)

Relied on persistent background pages that consumed memory constantly. Prone to security risks via remote code execution.

Modern (Manifest V3)

Uses ephemeral Service Workers. They wake up to handle events and terminate when idle. Enforces stricter Content Security Policies (CSP) preventing external script injection.

The "No Window" Challenge

Service Worker Constraints

Service workers don't have access to the DOM or the `window` object. This breaks many standard crypto libraries that rely on `window.crypto`.

The Solution: Polyfills

We implemented a custom `polyfill.js` to inject necessary globals into the Service Worker scope, allowing Ethers.js to function correctly in the background.

Cryptography 101

EtherVault employs a hybrid cryptographic approach:

  • Symmetric Encryption (AES-GCM): Used to lock the user's private key in local storage. The key is derived from the user's password.
  • Asymmetric Encryption (Elliptic Curve): Used for the wallet itself (Public Key for receiving address, Private Key for signing transactions).

The CryptoService Class

Key Derivation (PBKDF2)

We don't use the password directly. We derive a robust encryption key using PBKDF2 with 100,000 iterations and a random salt.

Encryption (AES-256-GCM)

This key encrypts the wallet data. GCM mode ensures both confidentiality (they can't read it) and integrity (they can't tamper with it).

async deriveKey(password, salt) { const enc = new TextEncoder(); const keyMaterial = await window.crypto.subtle.importKey( "raw", enc.encode(password), "PBKDF2", false, ["deriveKey"] ); return window.crypto.subtle.deriveKey( { name: "PBKDF2", salt, iterations: 100000, hash: "SHA-256" }, keyMaterial, { name: "AES-GCM", length: 256 }, false, ["encrypt", "decrypt"] ); }

Key Management Strategy

Zero-Knowledge Architecture

EtherVault servers (if we had them) never see the keys. The derived encryption key exists only in memory (RAM) while the user is logged in.

Session Timeout

To prevent memory dumping attacks, the key is wiped from memory after 15 minutes of inactivity, forcing a re-login.

The "Vault"

The "Vault" is a JSON object stored in chrome.storage.local.

It acts as an encrypted blob. Even if malicious software scans the browser's local storage files, they only see gibberish.

Only the specific extension instance with the correct password can turn this blob back into a usable private key.

HD Wallet Implementation

BIP-39 Mnemonics

We generate a 12-word seed phrase (e.g., "witch collapse practice feed..."). This is human-readable backup that can restore the entire wallet.

BIP-44 Derivation

From that single seed, we derive infinite accounts using the path m/44'/60'/0'/0/0. This standard ensures compatibility with MetaMask and Ledger.
